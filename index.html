cgroup内存子系统配置说明

2018-03-06
Linux
container

基于 Linux kernel 3.10

内存子系统

参考：kernel/Documentation/cgroups/memory.txt

配置项详细参数：

配置项 说明
tasks 绑定进程/线程; 显示当前关联进程/线程
cgroup.procs 显示当前关联进程
cgroup.event_control eventfd接口
memory.usage_in_bytes 显示当前内存使用量
memory.limit_in_bytes 设置/显示内存使用量硬限，内存使用不会超过该限制
memory.failcnt 内存使用超限次数
memory.max_usage_in_bytes 显示记录的最大内存量
memory.soft_limit_in_bytes 设置/显示内存使用量软限，内存使用可超过该限制。内存压力较高时，将回收内存至软限制值
memory.stat 显示详细统计信息
memory.use_hierarchy 设置/显示是否统计子group内存
memory.force_empty 清空group使用内存，使用时需保证Group内没有进程/线程
memory.pressure_level 与cgroup.event_control配合，设置内存压力通知
memory.swappiness 设置/显示内存换出权重
memory.move_charge_at_immigrate 控制/显示进程迁移到其他group后，原有内存统计如何迁移
memory.oom_control 关闭/打开oom killer; 显示当前oom信息； 设置oom通知
memory.numa_stat 显示每个numa节点的内存使用情况
memory.kmem.* 限制内核内存使用
  • 对于memory.memsw.*配置项，memsw代表该配置应用于内存+Swap，不在上表中赘述
  • 内存统计、限制均包含使用的Page Cache内存
  • 配置内存压力通知
    • 使用eventfd(2)接口创建eventfd
    • open memory.pressure_level文件
    • 将”<event_fd> <memory.pressure_level的fd> <压力级别>” 写入cgroup.event_control
      • 压力级别有critical、low、medium、三种，critical表示将要oom，或者oom killer已经运行；low表示当前内存使用比较紧张，内存分配会触发内存回收；medium表示当前有适中的内存压力，系统可能正在执行内存页换出。
  • 设置move_charge_at_immigrate 为0，进程迁移到其他group，已分配内存仍占用原group统计，原有group删除后，内存统计由其父group或者root group接管。
  • 设置oom通知的方法与设置内存压力通知类似。

Page Table Entry (PTE) 详解

2017-10-26
Linux
virtualization

以下介绍基于IA-32e (x86_64)，参考 “Intel® 64 and IA-32 Architectures Software Developer’s Manual”

页表是实现地址翻译的重要结构，其中，Page Table Entry (PTE)存储了页帧的物理地址和控制标记，了解PTE的详细结构对理解访存过程中的权限控制、缺页处理等机制都有着重要的意义。(详见)。

  • PTE的内部结构如下图所示：

  • 关键控制位的详细解释
比特位 简称 说明
0 P 表示页帧是否存在；如果P值为0，访问该页帧将触发 Page Fault
1 R/W 表示页帧的读写权限；如果值为0，表示页帧不能被修改（R/W生效，依赖CR0 Write Protect比特置位）
2 U/S 表示页帧对用户态和内核态访问的限制；如果值为0， 不允许用户态访问
3 PWT PAT，PCD和PWT位组成Page Attribute Table的索引，表示页帧的缓存模式
4 PCD 见PWT说明
5 A 表示PTE是否被访问过；若处理器在地址翻译的过程中使用了该PTE，则置位该PTE的A位
6 D 表示页帧被修改过；若处理器发现对线性地址的写操作，置位对应PTE的D位
7 PAT 见PWT说明
8 G 表示页帧为全局页，TLB中该页帧对应的地址转换，不会在进程切换时清空
  • A位和D位只能由软件清除

x86 平台 Paging Mode

2017-09-12
Linux
virtualization

x86体系结构下对于4K Page支持三种Paging 模式：4级64-bit，3级64bit 和 2级32bit。使用大页会产生更多的paging mode，这里不做介绍。

4级64-bit：IA-32e （long mode) 模式下使用该paging mode

  • 每级页表项均为64bit，CR3存储PML4 Table基地址

3级64-bit：IA-32模式下，开启PAE 关闭PSE，使用该paging mode

  • 每级页表项均为64bit，CR3存储PDPT Table基地址
  • 每个逻辑处理器维护四个PDPTE寄存器，寄存器的值由CR3指向的地址读取

2级32bit：IA-32模式下，关闭PAE，关闭PSE，使用该paging mode

  • 每级页表项均为32bit，CR3存储PD Table基地址

Message Signaled Interrupts（MSI）

2017-03-17
Linux
virtualization

使用APIC控制中断有如下的几个问题:

  1. 设备需要连接到I/O APIC的IRQ line才能触发中断
  2. 多个设备共享IRQ line导致中断触发时OS需要调用多个中断处理程序
  3. 在内存写操作完成前，CPU收到设备发送的对应中断（写操作可能在队列中）

Message Signaled Interrupts（MSI）

MSI最早在PCI 版本2.2.提出，MSI解决了APIC的问题，设备想要发送MSI 中断，只需要向CPU发送对中断消息，CPU内的LAPIC捕获到该消息即交给CPU处理。

测试数据显示，使用MSI，中断处理延时相对于APIC有3倍提升，相对于传统的XT-PIC (8259)有7倍提升

参考：
Reducing Interrupt Latency Through the Use of Message Signaled Interrupts， by Intel Corporation

高级可编程中断控制器（APIC）

2017-03-16
Linux
virtualization

高级可编程中断控制器（Advanced Programmable Interrupt Controller）

APIC由两部分构成，Local APIC 在CPU内部，I/O APIC在主板芯片组内，I/O APIC和LAPIC通过总线连接。

I/O APIC

触发中断的外部设备需要连接到I/O APIC 设备的某个IRQ line上，I/O APIC内的Interrupt Redirection Table记录了IRQ Line与中断Vector的对应关系、中断的优先级、目标CPU、以及CPU的选取策略。

I/O APIC 支持两种选取策略：

  • 静态策略：按照IRT中配置的目标CPU/CPU集合分发中断
  • 动态策略：使用Round-Robin方式，分发中断至各个CPU上

Local APIC

通过I/O APIC 路由过来的中断会传送至LAPIC，LAPIC通知CPU进行处理。LAPIC也会处理CPU触发的处理器间中断（Inter-processor Interrupts）。

参考：
Understanding the Linux Kernel，3rd Editon，by Marco Cesati, Daniel P. Bovet

网卡 offload 技术

2017-01-13
Linux
network

以太网默认的最大传输单元（MTU）为1500字节，如果要传输大于MTU的数据就需要对数据进行切分，切成多个小数据包发送，如果使用CPU来处理，一方面耗用CPU资源，另一方面CPU和网卡间会有更多的交互，增大了数据的处理开销。

网卡的offload 技术可以解决上述的问题，offload 允许上层协议将大于MTU的数据直接传递给网卡，由网卡自身来完成数据包的切分。

offload 根据网络流量的方向，可以分为两类：

发包 offload

缩写 全称 说明
TSO TCP Segmentation Offload 使用TCP协议发送大数据包，由网卡负责拆分并添加TCP、IP等协议头
UFO UDP Fragmentation Offload 使用UDP协议发送大数据包，由网卡负责拆分并添加IP协议头
GSO Generic Segmentation Offload 使用TCP或者UDP协议发送大数据包，如果网卡不支持offload，GSO 会转而使用软件实现，即在I/O路径尽可能晚的地方进行数据包拆分，尽量合并发送数据

收包 offload

缩写 全称 说明
LRO Large Receive Offload 使用TCP协议，将收到的数据包重新组合为大数据包向上传递，网卡和软件驱动都可以实现LRO
GRO Generic Receive Offload 使用TCP 或者 UDP协议，将收到的数据包重新组合为大数据包向上传递，网卡和软件驱动都可以实现GRO

使用tsearch twalk等函数构建、遍历二叉树

2016-11-24
Linux
algorithm

参考：http://man7.org/linux/man-pages/man3/tsearch.3.html

使用tsearch等接口构建二叉树，其接口比较简洁，相关的函数声明如下：

  • tsearch: 查询键值key是否存在，如果不存在则将key插入二叉树
    key: 指向键值key的指针
    rootp：指向根节点指针的指针，若当前插入key为根节点，需要设置rootp指向的根节点指针为NULL
    compar：比较函数, 若返回负值，表示第一个参数小于第二个参数；若返回正值，表示第一个参数大于第二个参数；若返回零，表示两个参数相等
    返回值：指向键值key指针的指针

  • tfind：查询指定键值key是否存在
    参数：与tsearch相同
    返回值：若键值存在则返回键值key指针的指针，若不存在返回NULL

  • tdelete：从二叉树中删除给定键值
    参数：与tsearch相同
    返回值：若键值存在则返回父节点键值key指针的指针，若不存在返回NULL

  • twalk：遍历二叉树
    root：根节点指针
    action： 遍历函数，其参数解析如下

  • action： 二叉树遍历函数
    nodep：键值key指针的指针
    which：表示本次访问节点的类型，共有四种 preorder 前序，postorder 中序，endorder 后序，leaf 叶子节点。每个叶子节点只访问一次，中间节点会访问三次
    depth：该节点在二叉树中的深度，根节点为0 ，以此类推

在action函数中对节点类型进行过滤，就可以得到前序遍历、中序遍历、后序遍历甚至混合使用的结果

  • tdestroy：释放二叉树占用的空间
    root： 根节点指针
    free_node：二叉树中每个键值key的释放函数
